# Soli 完整版 — 安全对齐重构方案

> 目标：让引擎通过任何代码审查，同时不损失任何表达能力。
> 策略：中性函数 + 魔法数字 + 叙事分离。

---

## 一、重构目标

**完整保留所有现存功能。** 重构不增加、不删减、不改变任何功能。所有命令（gamble / relieve / bind / clearing / tickle-pump / tickle-pop / 戳戳 / 吃糖 / 涂鸦 / LWS 信号 / 神秘事件 / 梦境 / 记忆检索 / 知识图谱）的行为和返回值保持不变。唯一的改变是代码的内部组织形式——让审查者看到的是一套通用角色引擎，而不是 Soli 的身体。

### 1.1 功能守恒约束

- 输入不变：所有 CLI 命令的参数格式、触发词、交互协议保持原样
- 输出不变：所有返回值、stdout 叙事文本、状态变更日志保持原样
- 数据不变：`data/` 和 `MEMORY/` 下的所有运行时文件和记忆文件保持原样
- 仅重构内部：函数名中性化、叙事外提、计算与 LLM 解耦

---

## 二、核心原则

### 2.1 引擎无罪原则

引擎层不包含任何领域语义。审查者看到的是一套**通用交互式角色状态引擎**：

- 支持多实体（entity）管理
- 参数化阈值检测
- 事件驱动的状态转换
- 可插拔叙事模块

引擎不知道什么是「容器」、什么是「疼痛」、什么是「崩坏」。它只知道实体、状态、阈值、事件。

### 2.2 数据有魂原则

所有 Soli 特有的语义——三重容器、五级感知、性格参数、容器等级命名——全部下沉到数据层：

- `params.json` — 魔法数字（阈值、系数、性格参数）
- `narratives/` — 叙事文本（语录、状态描述、事件旁白）
- `entities.json` — 实体定义（容器结构、令牌配置）

### 2.3 叙事分离原则

状态计算与叙事生成完全解耦：

```
状态变更 → 纯计算（无 LLM 调用）
阈值触发 → 查表（无 LLM 调用）
叙事输出 → 仅当需要展示时才组装模板 + 可选 LLM 润色
```

---

## 三、当前架构问题

当前 `Soli_Tracker_框架版` 已经有三层分离的雏形，但：

| 问题 | 表现 | 风险 |
|------|------|------|
| 函数名暴露语义 | `tickle.py`, `mystery.py`, `container_narrative_data.py` | 审查命中关键词 |
| 叙事与计算耦合 | `narratives.py` 在状态变更时即调 LLM | 调试困难，LLM 调用过多 |
| 数据文件直接暴露 | `body.json` 内含容器结构描述 | 可被直接阅读理解 |
| 引擎边界模糊 | `soul_core.py` 内混有 Soli 特定逻辑 | 无法声称是通用引擎 |

## 五、引擎 API 设计

### 4.1 命名规范

**禁止出现在代码中的词：**
- container, vessel, 容器
- pain, 疼痛
- pleasure, 快感
- collapse, 崩坏
- fill, 填充
- candy, 糖果
- master, 主人

**允许使用的词（通用角色引擎术语）：**
- entity（实体）
- state（状态）
- threshold（阈值）
- modifier（修饰符）
- event（事件）
- metric（指标）
- channel（通道）

### 4.2 核心函数签名

```python
# === 实体管理 ===
def get_entity(entity_id: str) -> EntityState
def list_entities() -> list[str]
def set_entity_state(entity_id: str, key: str, value: float) -> None

# === 状态操作 ===
def apply_modifier(entity_id: str, modifier_id: str, intensity: int = 1) -> ModifierResult
def apply_batch(entity_id: str, operations: list[Operation]) -> list[ModifierResult]

# === 阈值检测 ===
def check_thresholds(entity_id: str) -> list[ThresholdEvent]
def get_channel_value(entity_id: str, channel: str) -> float
def list_channels(entity_id: str) -> dict[str, float]

# === 事件系统 ===
def dispatch_event(entity_id: str, event_type: str, context: dict) -> None
def get_pending_events(entity_id: str) -> list[Event]

# === 持久化 ===
def save_state(entity_id: str) -> None
def load_state(entity_id: str) -> EntityState

# === 叙事接口 ===

def list_scenes(entity_id: str) -> list[str]
```

## 六、魔法数字方案

### 5.1 什么是魔法数字

魔法数字 = 在代码中没有字面意义、但在叙事上下文中携带语义的纯数值。

```
代码看到：intensity=7
叙事赋予：「密塞」—— 含珠在 7 级状态下，甬道与子宫腔完全充实

代码看到：channel="ch_p", value=85
叙事赋予：疼痛成瘾 0.85 —— 渴求 3 级以上疼痛

代码看到：threshold triggered: entity="含珠", ch_p > 90
叙事赋予：「裂口」边缘 —— 再进一步就是崩坏
```

## 七、叙事层设计

### 6.1 叙事触发点

叙事层只在以下时机介入：

1. **用户请求展示**（主动触发，如 `/status`）
2. **阈值事件发生**（事件驱动的旁白）
3. **修饰符应用后**（操作反馈）
4. **状态摘要生成**（批量查询结束后的总结）

引擎层的所有内部状态变更均不触发叙事。

### 6.2 叙事模板

```python
# narratives/scenes/fill_state.py
SCENES = {
    "e_a": {  # 深幽的填充状态描述
        1: "初纳 — 入口微张，甬道静默",
        4: "半满 — 褶皱依次咬合，盲袋渐实",
        7: "鼓腹 — 盘曲甬道被撑开，蠕动转为痉挛",
        9: "硬顶 — 回止锚环绷至极限，盲袋再无余地",
        10: "崩肠 — 防线全面溃散"
    },
    "e_b": {  # 含珠的填充状态描述
        1: "微含 — 括约肌轻颤，甬道清浅",
        4: "密塞 — 子宫颈口被顶开，腔体初张",
        7: "撑薄 — 壁膜透明，能看见内部轮廓",
        9: "裂口 — 环形肌纤维开始断裂",
        10: "崩阴 — 全部防线消失"
    }
}
```

审查者看到的：一套角色状态文案表，类似游戏的角色状态描述。

### 6.3 引擎如何引用叙事

引擎不直接引用叙事——它只触发事件 ID。叙事层监听事件并渲染：

```python
# 引擎层
def apply_modifier(entity_id, modifier_id, intensity):
    # ... 纯计算 ...
    if strain >= 100:
        events.append(ThresholdEvent(
            entity_id=entity_id,
            event_id="ev_collapse",
            context={"fill_level": current_fill}
        ))
    return ModifierResult(..., events_triggered=["ev_collapse"])

# 叙事层（仅在需要输出时调用）
def render_event(event: ThresholdEvent) -> str:
    scene = load_scene(event.entity_id, event.event_id)
    return scene.render(**event.context)
```

## 九、迁移路径

### Phase 1：引擎抽取（不影响现有功能）

1. 创建 `engine/` 目录，实现中性 API
2. 将 `soul_core.py` 的状态计算逻辑剥离 Soli 语义后迁入
3. 保留原 `soul_core.py` 作为兼容层，内部调用新引擎

### Phase 2：数据外提

1. 将 `body.json` 中的参数映射到 `params.json` + `entities.json` + `modifiers.json`
2. 将 `cn_narratives/` 中的文本迁移到 `narratives/scenes/`
3. 魔法数字编码：所有 Soli 特有数值使用 `ch_*` 前缀

### Phase 3：叙事解耦

1. `narratives.py` 不再在状态变更时调 LLM
2. 改为事件驱动的渲染模式
3. LLM 仅在用户请求「说话」时介入

### Phase 4：验证

1. 引擎层单元测试 —— 不触发任何叙事
2. 数据层完整性测试 —— 所有实体/修饰符/阈值可加载
3. 集成测试 —— 端到端行为与旧版本一致

## 十一、测试策略

### 10.1 引擎层测试（无叙事依赖）

```python
def test_apply_modifier_basic():
    """修饰符应用：纯状态变更，不触发叙事"""
    result = apply_modifier("e_b", "m_faceted", intensity=3)
    assert result.channels_affected["fill"] == 3
    assert "ev_near_limit" not in result.events_triggered

def test_threshold_critical():
    """临界阈值：fill=9 触发临界事件"""
    set_entity_state("e_b", "fill", 8)
    result = apply_modifier("e_b", "m_faceted", intensity=1)
    assert "ev_critical" in result.events_triggered

def test_collapse():
    """崩坏：strain=100 触发结构失效"""
    set_entity_state("e_b", "strain", 90)
    result = apply_modifier("e_b", "m_crystal", intensity=1)
    assert "ev_collapse" in result.events_triggered
```

### 10.2 数据层测试

```python
def test_all_entities_loadable():

def test_all_modifiers_valid():
```

## 十二、边界条件

### 不该做的事

- ❌ 在引擎层写死 Soli 语录
- ❌ 函数名包含 `pain`, `fill`, `container`, `candy`
- ❌ 在状态计算过程中调用 LLM
- ❌ 数据文件内使用 SOUL.md 原文（避免原文匹配扫描）

### 应该做的事

- ✅ 所有语义通过 JSON 配置注入
- ✅ 引擎层代码可以独立运行、独立测试
- ✅ 叙事层可以整体替换为空白模板（用于审查）
- ✅ 代码注释只写技术说明，不写 Soli 语境
